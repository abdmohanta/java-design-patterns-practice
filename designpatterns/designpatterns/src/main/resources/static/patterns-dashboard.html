Pattern Map

SingletonPattern

See source: com.debasish.designpatterns.creational.SingletonPattern

Definition: Ensures a class has only one instance and provides a global point of access to it. Useful for shared resources like configuration, caches, or connection managers.

PrototypePattern

See source: com.debasish.designpatterns.creational.PrototypePattern

Definition: Create new objects by copying a prototypical instance, allowing efficient object creation when construction is expensive or complex.

FactoryPattern

See source: com.debasish.designpatterns.creational.FactoryPattern

Definition: Encapsulates object creation logic in a factory class or method, letting callers request objects without knowing concrete classes; simplifies instantiation and promotes loose coupling.

BuilderPattern

See source: com.debasish.designpatterns.creational.BuilderPattern

Definition: Provides a fluent API to construct complex objects step-by-step, separating construction from representation and improving readability for many-parameter objects.

AbstractFactoryPattern

See source: com.debasish.designpatterns.creational.AbstractFactoryPattern

Definition: Offers an interface for creating families of related objects without specifying concrete classes, ensuring consistent combinations of product variants.

AdapterPattern

See source: com.debasish.designpatterns.structural.AdapterPattern

Definition: Converts one interface into another expected by clients, allowing incompatible classes to work together without changing their source code.

CompositePattern

See source: com.debasish.designpatterns.structural.CompositePattern

Definition: Compose objects into tree structures to represent part-whole hierarchies, treating individual objects and compositions uniformly to simplify client code.

DecoratorPattern

See source: com.debasish.designpatterns.structural.DecoratorPattern

Definition: Attach additional responsibilities to objects dynamically by wrapping them with decorator objects, providing flexible behavior extension without inheritance.

FacadePattern

See source: com.debasish.designpatterns.structural.FacadePattern

Definition: Provide a simplified higher-level interface that hides the complexity of a subsystem, making it easier for clients to interact with multiple components.

ProxyPattern

See source: com.debasish.designpatterns.structural.ProxyPattern

Definition: Provide a surrogate or placeholder for another object to control access, add caching, lazy initialization, or access control without changing the original object.

CommandPattern

See source: com.debasish.designpatterns.behavioral.CommandPattern

Definition: Encapsulate requests as objects (commands) to parameterize clients, support queuing, undo/redo, and decouple command issuance from execution.

ObserverPattern

See source: com.debasish.designpatterns.behavioral.ObserverPattern

Definition: Define a one-to-many dependency so that when an object changes state, its dependents are automatically notified and updated, enabling event-driven designs.

StatePattern

See source: com.debasish.designpatterns.behavioral.StatePattern

Definition: Allow an object to alter its behavior when its internal state changes by encapsulating states as separate objects and delegating behavior to them.

StrategyPattern

See source: com.debasish.designpatterns.behavioral.StrategyPattern

Definition: Define a family of interchangeable algorithms, encapsulate each one, and make them interchangeable so clients can select algorithms at runtime.

TemplateMethodPattern

See source: com.debasish.designpatterns.behavioral.TemplateMethodPattern

Definition: Define the skeleton of an algorithm in an operation, deferring some steps to subclasses so they can vary certain parts without changing structure.

CircuitBreakerPattern

See source: com.debasish.designpatterns.resilience.CircuitBreakerPattern

Definition: Protect downstream services by stopping calls to a failing service for a period (OPEN), probing it (HALF-OPEN), and closing when healthy again to prevent cascading failures.

README

Resilience Patterns This directory contains demo implementations for common resilience/fault-tolerance patterns. Files: - `CircuitBreakerPattern.java` - toy circuit breaker local implementation. Expected output: shows state transitions from CLOSED -> OPEN -> HALF_OPEN -> CLOSED depending on failures and open duration. The demo prints calls and failure counts. - `RetryPattern.java` - simple retry policy with exponential backoff and jitter. Expected output: attempts with retry delays and eventual success (or final failure after max attempts). - `TimeoutPattern.java` - uses an ExecutorService to run tasks with a timeout. Expected output: first case completes within timeout; second case times out and is cancelled. - `Resilience4jExamples.java` - a short demo that demonstrates presence of Resilience4j on the classpath. For production examples, the pom includes explicit Resilience4j modules (`resilience4j-core`, `resilience4j-circuitbreaker`, `resilience4j-retry`, `resilience4j-timelimiter`). Notes: - These examples are intentionally minimal and self-contained for learning. They are safe to run locally and print traces to stdout. - For production use, prefer using a mature library like `resilience4j` and integrate it with metrics collection and configuration. Running tests: - The repository includes basic JUnit tests that run each demo's `main` method. Use `mvn test` to run them (requires internet to download test dependencies).

RetryPattern

See source: com.debasish.designpatterns.resilience.RetryPattern

Definition: Retry transient operations with a configurable backoff and optional jitter to handle temporary failures while avoiding thundering herds; ensure retries are safe for idempotent operations.

TimeoutPattern

See source: com.debasish.designpatterns.resilience.TimeoutPattern

Definition: Bound operation duration to fail fast and free resources; useful to prevent blocked threads and propagate deadlines in distributed systems.
